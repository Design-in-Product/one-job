## Innovations worth cross-pollinating

- **In-progress as emergent structure, not status**: the parent card is
  "in progress" exactly when its interior isn't finished — a state
  machine where the state is *readable from the containment tree*
  rather than stored. Products drowning in status enums could steal
  this: most "statuses" are derivable facts.
- **The safety-net pattern for local-first apps**: meta + dated
  snapshots + quarantine-don't-clobber + restore-on-missing, ~60 lines,
  test-first. Any sibling holding user data in localStorage should have
  exactly this layer between "the browser lost it" and "the user lost
  it."
- **Dogfooding as capture channel**: Xian keeps product notes as cards
  in the product — the feedback loop IS the usage loop.

### Build #2 shipped — Undo toast (RED ZONE, test-first)

*What*: after complete or defer, the toast carries a 5-second Undo
action; tapping it puts the card back exactly as it was (position,
completion, deferral count).

*How, and the design lesson in it*: undo is implemented as **snapshot
restore, not inverse operations**. Before mutating, the handler deep-
clones the task (`structuredClone` — necessary because the local store
mutates the very objects React state holds; a shallow reference would
"snapshot" the already-mutated task). Undo = hand the snapshot back to
a new store method `restoreTask(snapshot)`. One method undoes *any*
mutation, past or future, instead of one inverse per operation —
snapshot-based undo scales where command-based undo multiplies.

*The honesty seam*: `restoreTask` is optional on the TaskStore
interface. Local and demo stores implement it; the API store doesn't
(the backend can't restore sort order atomically), so the UI simply
doesn't offer Undo there. Same pattern as `importTasks` — capability
discovery over feature flags: the UI asks "can this store keep the
promise?" rather than "which mode am I in?"

*Verification*: 3 new store tests written FIRST (un-complete restores
cleanly; deferred card returns to top; ghost snapshot throws), then
Playwright drove the real flow: swipe right → toast → tap Undo → card
back on top, un-completed. 22 tests green.

### Build #4 shipped — Portal crispness

*Diagnosis worth teaching*: the "app swimming in an ill-fitting
portal" feel was **three mismatched materials**, not a layout bug —
browser/PWA chrome painted brand coral (`theme-color: #f35343`), the
PWA splash painted cream (`background_color: #FFF7ED`), and overscroll
rubber-banding revealed a third body color behind the app. Each
boundary betrayed a frame. Fix: every surface (html, body, theme-color,
manifest splash) now shares the app's own top color (gray-50 #f9fafb),
and `overscroll-behavior-y: none` removes the rubber-band reveal.
Computed styles verified in Chromium; the *felt* difference needs a
real iPhone — batched for Xian's review.

*Cross-pollination lesson*: web apps feel "inside a portal" when the
chrome colors are treated as brand real estate instead of app surface.
theme-color is not a logo slot — it's the material the app chrome is
made of. (The brand still owns the card faces, the icon, the splash
logo — losing chrome-coral costs nothing.)

### Build #6 shipped — Domain layer extraction (the R1 stepping stone)

`src/domain/tasks.ts` now holds the rules — reviveTask, sortTasks,
nextSortOrder, applyCompletion, applyDeferral — as pure functions with
their own 7-test suite pinning today's exact behavior (sort tie-breaks,
missing-field defaults, deferral counting). LocalTaskStore delegates to
them; it keeps only storage concerns (persistence, snapshots, undo
restore). Zero visible change, proven three ways: the 22 pre-existing
store tests ran unmodified over the refactored code, tsc/build clean,
and the undo flow re-driven in Chromium behaved identically.

*Why this ordering matters (the teachable part)*: the R1 migration will
change what the rules ARE. Extracting them first — under an unchanged
test harness — separates "move the rules" from "change the rules" into
two independently-verifiable steps. When schema v2 lands, it edits one
pure module with focused tests, not a storage class with side effects.
One deliberate exception kept: substack-task completion stays inline in
the store because it *doesn't* set `status` the way main-task
completion does — extracting it into applyCompletion would have
silently changed behavior. Refactors preserve bugs-and-quirks too;
that's what makes them refactors.
