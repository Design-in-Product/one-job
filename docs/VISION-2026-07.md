## Item 1 — Subtasks, not substacks

**Xian, 2026-07-03 evening (verbatim gist):** The substack concept is
overly complicated right now. The missing concept is that **any task
should be able to have subtasks if you want them** — tasks within tasks,
supported ad infinitum. Creating a *stack* inside a task (a named
container) is actually a next-level thing for much more complicated
projects, will be much rarer, and doesn't necessarily need to be a
feature we support right now. We can worry about substacks later.

**Code-level context (no steering, just facts):**
- Today's model is the opposite shape: a task cannot have child tasks
  directly; it can only have named `Substack` containers
  (`src/types/task.ts`: `Substack { id, name, tasks: Task[] }`), and
  tasks inside a substack cannot themselves have anything.
- So the current model is *task → named stack → flat tasks*, exactly the
  "rare, next-level" construct, made mandatory as the only door to any
  nesting — and depth-limited to one level.
- The vision model is *task → tasks → tasks…* (recursive, optional,
  unnamed), with named stacks as a later, rarer layer on top if ever.
- Happy accident: the big-picture audit's "substack semantics are
  undefined" gap (what happens when all substack tasks complete, etc.)
  largely dissolves under this model — the semantics question becomes
  the simpler "how do a parent and its subtasks relate," asked once,
  recursively.

## Item 2 — Done is a stack, not a dead end (and so are Archive and Trash)

**Xian:** From the **Done stack**, a **left swipe marks a task undone**
(back to the main deck); a **right swipe archives it**. There's an
**Archive stack** where, again, you can move a task back to Done or move
it onward to **deleted/Trash**.

**The shape this implies:** a task lifecycle laid out as a chain of
stacks — *Todo → Done → Archive → Trash* — where **right-swipe always
advances a card along the chain and left-swipe always moves it back**.
The same two gestures the user already knows, gaining meaning from
*which stack you're in* rather than from new UI.

**Code-level context:**
- Today Completed is a display-only list view (`CompletedTasksView`),
  not a stack: no gestures, no un-complete, no archive, no delete. It
  was gap #3 in the big-picture audit ("Completed is a dead end"), and
  gap #2 (no delete UI) also resolves inside this chain — deletion is
  just the far end of the conveyor rather than a destructive button.
- One asymmetry worth a clarifying question when we discuss (not now):
  in the Todo deck, left-swipe currently means *defer within the stack*
  (to bottom), not *move back a stage* — the only stack where left
  isn't "regress along the chain."

## Item 3 — Stacks live side-by-side on a canvas

**Xian:** These stacks sit **next to each other on a canvas**. Generally
you're focused on one of them, but you should be able to **back up
(zoom out) and swipe over to another one**. And the canvas is where
*multiple top-level decks* live too — the stack for all your health
stuff and the one for all your work stuff might sit on that top-level
canvas.

**What this unifies:** Done/Archive/Trash (Item 2) aren't modal views —
they're neighbors on the same canvas as the Todo deck(s). Navigation is
spatial (zoom out, pan, zoom in) rather than menu-driven. Multiple
decks-by-life-area is the same mechanism, not a separate feature.

**Code-level context:**
- This is the return of the original spatial seed: the 2025 Card Deck
  pivot spec described "spatial thinking" as the foundation, and the
  long-press arc menu was always a stand-in. The audit's gap #6
  ("long-press is the only door to Completed/Settings") is a symptom of
  the views not yet having a *place*.
- Today the app has exactly one deck (one flat task list per storage
  key) and modal views for everything else. Nothing in the TaskStore
  seam assumes one deck, but nothing supports several either.

## Item 4 — Text entry is the wrong tool for this interface (ergonomic concern)

**Xian:** It's going to be very cumbersome to enter text and manipulate
it this way. This might connect to the ideas about populating and
managing the deck automatically (integrations, agents). But — again —
we have to get the basics, the plumbing, and the ergonomics right first
**in the concept model as well** as the code.

**Capture note:** this is registered as a *tension to design around*,
not a feature request: the card interface is optimized for triage
(swipe, flip, focus), and authoring/editing text on it fights the
grain. The auto-population ideas (MCP inbox memo, adapters) are one
possible relief valve but remain parked; the concept model comes first.

## Item 5 — The text should fill the card ✅ (shipped 2026-07-03 evening)

**Xian:** Purely cosmetic, but strongly felt: the card is mostly empty
space — a conservative title above a large, almost-always-empty
description area. Title and description should keep their relative
proportions (description smaller) but be **sized to fit**: bumped up
until they essentially wrap and fill the whole available space — "a nice
big bold statement, as big as it can be inside the card shape."

**Status:** implemented same evening (small, self-contained, purely
presentational — didn't touch the concept model). `useFitText` hook
binary-searches the largest font size at which title + description
(description at 0.6× the title) still fit the card's content box;
re-measures on resize. Verified via Playwright at mobile viewport for
both extremes (3-word title alone; long title + long description).

## Item 6 — Zoomed in, the deck IS the UI

**Xian:** In some cases, zoom in so the deck of cards is the *entire*
UI, aside from the phone chrome — as if the phone in your hand is
literally a pack of cards, rather than a screen showing a thing that
looks like a pack of cards. With, of course, the ability to pinch and
zoom back out and see the deck sitting on top of the invisible canvas.
"More of the kind of way I would like it to feel when I use it in my
hand."

**Capture note:** this is the zoom continuum of Item 3's canvas — the
innermost zoom level. Canvas (all stacks) ⇄ one stack framed on the
canvas ⇄ the card filling the screen edge-to-edge. Item 5 (text fills
the card) and this (card fills the phone) are the same instinct at two
scales: no dead space between you and the task.

## Item 8 — A card is a container: the flipped card's face is a canvas

**Xian (reconciling Items 1 and 3):** Here's where substack and subtask
got confused, and the reconciliation may be easier than we think. In a
normal list app, a subtask is another checkbox list under an opened
item. But the whole concept of One Job is **one task at a time — even if
it's a subtask, it's one subtask at a time**. So the moment you create a
subtask, you're creating **a card within a card**: the face of the card
you flipped over is now a *canvas*, and that canvas holds a smaller deck
of cards — at first containing just the one task you added. You add
tasks the exact same way you always do. The card is a container.

**What this dissolves:** subtasks and the canvas aren't two features —
they're one recursive structure. Canvas holds decks; deck holds cards;
a card's interior is a canvas. The named "substack" construct isn't
needed for nesting at all (it was never the missing concept — the
container was). One-card focus survives at every depth.

**Gesture reasoning captured with it:**
- *Why doesn't left-swipe at the top level go "up a level"?* Because
  there is no higher deck — above the top-level deck is only the
  canvas, and a card doesn't go to the canvas (cards belong to decks;
  decks sit on the canvas). So left = to the bottom of this deck, "and
  that's simply the paradigm that we have."
- *Lower decks.* ~~Flagged for re-confirmation~~ **RESOLVED by Item 10:
  subtask decks behave exactly like every other deck — left-swipe defers
  to the bottom of that stack, at any depth. The special "popping back
  up" mechanics belong only to the reserved lifecycle decks
  (Done/Archive/Trash), which are a distinct kind of deck entirely.**
- *Promoting a subtask up a level* is a different act and would need a
  different gesture — maybe a vertical swipe toward the top of the
  screen. Explicitly not designed now.

**Code-level context:** today the flip has exactly two faces: CardBack
(the deck design) and TaskCard (title/description). Under this model
the task face gains a second life as a container view when subtasks
exist. Nothing in the TaskStore seam prevents `Task → tasks[]`;
the `Substack` layer is what this supersedes.

## Item 9 — The portal problem: dead space, and what it's made of

**Xian (complicating Item 6):** The phone is an elongated shape, with
chrome at the top — and even now the image sometimes **swims inside an
ill-fitting portal**, a thing web-based apps are prone to: the app
appears to be physically *inside* a badly fitted frame rather than
crisply aligned with the viewport. Two parts to this:

1. **A fit-and-finish defect to tune eventually**: the swimming/portal
   feel itself — content should sit crisp against the viewport edges,
   not float in a frame.
2. **A design question about the dead space**: a 5:7 card (or deck) on
   a ~19.5:9 screen leaves real estate no card can fill. What *is*
   that space? "Does it sort of look like empty space? Is it black? Is
   it white? Is it a reflection? Is it abstract?"

**Capture note:** Item 6 (phone-as-pack-of-cards) makes this question
unavoidable — if the deck is the entire UI, the material around the
deck IS the rest of the UI. The dead space stops being leftover and
becomes the table the deck is sitting on (felt/void/reflection/
abstract — a designed surface either way). Question is open; no answer
chosen.

**Code-level suspects for the "swimming" (for the eventual tuning, not
now):** body background vs. app background mismatch showing through at
the edges; iOS overscroll rubber-banding revealing the page behind the
app; `theme-color` meta and PWA manifest `background_color` not
matching the app surface; dvh/vh fallback seams; safe-area insets
padding the shell but leaving the underlying background exposed. All
checkable when we do the pass.

## Item 10 — Reserved lifecycle decks are their own kind of thing

**Xian (correcting Coral's over-read of Item 8):** Subtasks have the
same mechanics as tasks — they behave *exactly* the same way; swipe
left and you go to the bottom of that stack, at any depth. What's
actually special is the **reserved "afterlife" decks** — Done, Archive,
Trash, or whatever we decide that paradigm holds. *Those* are the decks
with the moving/popping-back-up-the-stack mechanism (Item 2's chain:
left regresses, right advances). **Any deck can feed into them** when a
card is designated done. They may not even be presented differently in
the UI — but they must stay **distinct as a concept** from a subdeck /
substack / subtask. (Xian: "I don't know the right sort of computer
science concept for those especially reserved lifecycle decks.")

**The clean two-kind model this leaves:**
- **Working decks** — the top-level decks on the canvas and every
  card's inner deck, uniform mechanics at every depth (right =
  complete, left = to the bottom of this deck). One recursive kind.
- **Lifecycle decks** — a small, fixed, system-owned chain
  (Done → Archive → Trash) that all working decks at all depths drain
  into; cards there move along or back up the chain rather than
  cycling. Not part of the recursion — a card's interior contains
  working decks only.

**Naming candidates from prior art (offered, not chosen):** email
solved exactly this split — user folders vs. *special-use mailboxes*
(IMAP's \Trash, \Archive, \Sent; RFC 6154); filesystems and photo apps
call theirs *system folders/albums* vs. user ones. In state-machine
terms the lifecycle chain is the task's state machine *reified as
places* — each afterlife deck is a state you can stand in and look
around. Working shorthand until Xian picks: **system decks** vs.
**user decks**.

**Provenance corollary — CONFIRMED by Xian ("good catch yes!"):** since
every working deck at every depth drains into the same Done, a card
arriving there remembers its home deck; left-swipe from Done returns it
*home*, not to a generic top deck. Cards carry provenance.

## Item 11 — The 2.0 thesis: an experience layer over sunk-cost backends

**Xian (verbatim gist):** Plugging into existing task backends —
**import first, then two-way sync** — and having a way of **preserving
the value-added concepts on the One Job layer while federating and
normalizing the metadata from connected services (and preserving the
mapping both ways)** — is critical as a 2.0 release. The app will
either be **(a) a toy, or (b) a full-featured task/project-management
tool, until it is willing to just be a better experience layer** on the
person's (or their employer's) already-sunk-cost chosen task manager.
It might be nice if it can pull from multiple sources — your Asana,
your Trello, your Todoist, what have you.

**Why this is the strategic keystone:** it names the third path out of
a losing fork. Competing on features loses to Asana; staying
standalone caps the product at "toy." The identity that scales is:
*your organization's task system already exists; One Job is how a
human faces it, one card at a time.* Requirements it pins down:
1. **Import before sync** (Gall's law applied to integration — read-only
   first, prove the mapping, then earn write access).
2. **The One Job layer is preserved value**: deck order, deferral
   history, subtask-canvas structure, lifecycle placement — concepts
   the source services don't have must survive sync rather than being
   flattened into them.
3. **Bidirectional mapping is a first-class artifact** (One Job id ⇄
   source id, field correspondences, both directions preserved).
4. **Multi-source federation**: several backends normalize into one
   deck paradigm side by side (which the canvas, Item 3, gives a
   natural home — a deck per source is one possible shape).

**Code-level context:** the TaskStore seam was built for exactly this
moment; `source`/`externalId` fields already exist on Task (vestigially
— provenance from Item 10 generalizes them); the old Integration domain
docs assumed push-export, which this reverses (One Job as the *view*,
not the *exporter*).

## Item 13 — Updates must never cost the user their deck 🔴 (release blocker; safety net shipped 2026-07-04)

**Xian (using the app as his daily driver):** Right now every pushed
update wipes his local data — "not something that we can release." We
need a concept of backing that data up, exporting it, and preserving it
whenever an update could wipe it.

**Status:**
- **Safety net shipped same day** (test-first, 6 new store tests):
  every save now also writes a `.meta` record (count + timestamp) and a
  dated `.snapshot.YYYY-MM-DD` copy (7-day retention). If the main key
  comes up missing but meta says data existed, the store silently
  restores the newest readable snapshot. Corrupt data is quarantined to
  a `.corrupt.<ts>` key instead of being overwritten (the old code
  clobbered it — fixed). An empty deck can never overwrite a non-empty
  snapshot. Verified end-to-end in Chromium: create → delete key →
  reload → card returns.
- **Root cause still open** — nothing in the web code path can wipe
  `oneJobTasks` on deploy, which suggests the runtime matters (native
  build reinstalls wipe the app container; iOS can evict site data for
  non-installed Safari tabs; dev-server port changes move the origin).
  Xian's recollection (07-04): possibly pre-TestFlight, directly in the
  browser — consistent with Safari-tab eviction or an origin change.
  Watching for recurrence; meta records now give forensics if it does.
- **Leading hypothesis (07-04 late night)**: **PWA reinstall wipes.**
  On iOS, deleting and re-adding a home-screen web app destroys its
  storage. The deploy pipeline was silently broken (Node-18/crypto),
  so updates never arrived → the natural user move to "get the update"
  is delete + re-add → data gone. This would explain why wipes
  *correlate with updates*: reinstalls correlate with updates. If
  confirmed, the countermeasures are exactly what's now shipped
  (working pipeline, backup nudge, export) plus a standing rule:
  **never delete the icon to update — the icon IS the database.**
- **Remaining design work**: export nudges ("last backup N days ago"),
  and eventually off-device durability (R3/R4 server era). Manual
  export/import via Settings already exists.

## Item 14 — Deal a card to an agent

**Xian:** A natural extension once agents are wired in: look at a card
and **assign it to an agent** — "I don't want to do this task. I think
Pard could do this. Carl could do this." Because the agent is known to
the system, the task gets dispatched; when the agent completes it, it
reports the task done. Mental model: **the card goes onto the agent's
deck** — while staying visible on the human's deck, noted as in
progress.

**And the in-progress question self-resolved mid-thought:** One Job
deliberately lacks "in progress" — cards are binary, done or not. Xian's
resolution: that's *still true at leaf level*, and **in-progress is an
emergent property of the interior** — a card whose inner stack isn't
finished is in progress; it comes back to the top and you *see* there's
more work. No status flag needed; when interiors exist, all cards
remain binary and "in progress" is just "has an unfinished inside."

**Domain-model resonances (noted, not designed):**
- *Decks belong to actors.* Assignment = dealing a card onto another
  actor's deck — the same move-between-decks mechanic as everything
  else; "state is place" extends to *whose place*.
- An assigned card on your deck is conceptually a card whose interior
  fate is in someone else's hands — the delegation marker and the
  unfinished-interior marker may be the same visual language.
- This is the **outbound** direction of the agent story: R4's MCP inbox
  was agent→human (agent deals you a card); this is human→agent (you
  deal the agent a card, and completion events flow back). Two-way
  card-dealing over one seam.

## Item 15 — Completing a parent with an unfinished interior: not allowed, with a rescue flow

**Xian (2026-07-04, explicitly "not necessarily the final design... a
pathway through the possibility space we can explore together"):**
It should **not be allowed** — but the refusal must not be annoying,
because sometimes it's a technicality. The intuition, in sequence:

1. You swipe a parent card right, believing it done — but it has
   unfinished sub-tasks you didn't notice (which itself "should be
   easier to tell": a finished interior shouldn't be visible at all —
   when a sub-deck is all done, it shouldn't be there anymore).
2. The card **bounces back** — it doesn't complete; it comes back into
   view.
3. The **sub-tasks come into focus**: the top of the sub-deck shows,
   revealing what's still open.
4. At that moment, each unfinished sub-task offers two moves:
   - **swipe right** = "this is your chance to say it's done" — clear
     it;
   - **swipe left** = *in this moment only*, instead of defer-to-
     bottom, **escalate it to a first-class task** of its own (it
     leaves the interior and joins the parent's deck).
5. With several stragglers, you make that choice for each; when the
   interior empties, the parent completes.

**Design principles embedded (worth keeping even if the flow changes):**
- The refusal is physical (bounce-back), not verbal (no dialog).
- The block is also the reveal — the interface answers "why not?" by
  showing you, not telling you.
- Left-swipe stays "not now," but context upgrades its meaning:
  during the rescue it means "not *here*" — promotion instead of
  deferral. Two gestures, meaning from the moment.
- Empty interiors vanish — a card with a finished sub-deck reads as
  childless, so bounce-backs are never technicalities about work that
  is actually done.

**Related answers recorded at the same time:**
- **Trash purge policy**: a preference with a smart default — Xian
  dislikes offloading choices to users; default TBD in discussion.
- **Deck-depth signal (build #5, still open)**: he's exploring an
  embodied-cognition / spatial-intelligence direction rather than
  choosing number-vs-shadow — maybe the card *back* differs with
  depth: a darker shade, "these are in the shadow, the underworld."
  He'll play with the app and look for a physical/gestural/visual
  affordance before deciding. Not blocked-on-Coral; blocked on play.

## Item 17 — Named sub-decks: one now, many later (premium + compatibility)

**Xian (2026-07-05 evening, sharpening Item 8):** Build the subtask-
deck concept as: a subtask is a card in a **sub-deck**. Each card can
set up **one sub-deck for now**, but the **domain model should allow
for more, distinguished by names** — exactly parallel to multiple
top-level decks on the default canvas (work, life, …). Multiple named
sub-decks are possibly a **future premium feature**, and they increase
**compatibility with tools that support multiple subtask lists** (the
2.0 federation angle: Asana sections, Trello checklists, etc. map onto
named sub-decks instead of being flattened).

**What this resolves:** the domain model's deliberately-unresolved
question ("can an interior canvas hold multiple decks?") now has its
answer — *the model says yes and always did (canvas holds decks); the
UI ships one until the rare case earns its surface.* The old Substack
NAME concept returns home as the name of an interior deck. Monetization
note recorded: depth is free, breadth may be premium.

## Item 22 — Cards age like library card stock (the subtle nag)

**Xian (2026-07-06):** Cards should appear to physically **yellow and
age**, like old card stock in a library — a subtle nag. :D

**Capture notes:**
- The data already exists: `createdAt`, `deferredAt`, `deferralCount` —
  age and neglect are computable today; this is pure material rendering
  (a slow tint/texture curve on the card face).
- It belongs to the same design family as the deck-depth "darker backs,
  shadow, underworld" exploration: **time and depth rendered as
  material, not numbers**. The two should be designed together as the
  app's material language (with Item 7's custom card backs as the third
  member).
- The emotional design is the point: a checklist nags with red badges
  and overdue counts (guilt by arithmetic); a yellowing card nags the
  way an object does — quietly, physically, and only when you look at
  it. Anti-guilt pressure.
- Design questions parked for the material conversation: does deferral
  age a card faster than time? Does aging reset on edit (handling the
  card renews it)? Do Done cards stop aging (finished things rest)?
