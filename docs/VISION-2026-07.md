## Item 1 — Subtasks, not substacks

**Xian, 2026-07-03 evening (verbatim gist):** The substack concept is
overly complicated right now. The missing concept is that **any task
should be able to have subtasks if you want them** — tasks within tasks,
supported ad infinitum. Creating a *stack* inside a task (a named
container) is actually a next-level thing for much more complicated
projects, will be much rarer, and doesn't necessarily need to be a
feature we support right now. We can worry about substacks later.

**Code-level context (no steering, just facts):**
- Today's model is the opposite shape: a task cannot have child tasks
  directly; it can only have named `Substack` containers
  (`src/types/task.ts`: `Substack { id, name, tasks: Task[] }`), and
  tasks inside a substack cannot themselves have anything.
- So the current model is *task → named stack → flat tasks*, exactly the
  "rare, next-level" construct, made mandatory as the only door to any
  nesting — and depth-limited to one level.
- The vision model is *task → tasks → tasks…* (recursive, optional,
  unnamed), with named stacks as a later, rarer layer on top if ever.
- Happy accident: the big-picture audit's "substack semantics are
  undefined" gap (what happens when all substack tasks complete, etc.)
  largely dissolves under this model — the semantics question becomes
  the simpler "how do a parent and its subtasks relate," asked once,
  recursively.

## Item 2 — Done is a stack, not a dead end (and so are Archive and Trash)

**Xian:** From the **Done stack**, a **left swipe marks a task undone**
(back to the main deck); a **right swipe archives it**. There's an
**Archive stack** where, again, you can move a task back to Done or move
it onward to **deleted/Trash**.

**The shape this implies:** a task lifecycle laid out as a chain of
stacks — *Todo → Done → Archive → Trash* — where **right-swipe always
advances a card along the chain and left-swipe always moves it back**.
The same two gestures the user already knows, gaining meaning from
*which stack you're in* rather than from new UI.

**Code-level context:**
- Today Completed is a display-only list view (`CompletedTasksView`),
  not a stack: no gestures, no un-complete, no archive, no delete. It
  was gap #3 in the big-picture audit ("Completed is a dead end"), and
  gap #2 (no delete UI) also resolves inside this chain — deletion is
  just the far end of the conveyor rather than a destructive button.
- One asymmetry worth a clarifying question when we discuss (not now):
  in the Todo deck, left-swipe currently means *defer within the stack*
  (to bottom), not *move back a stage* — the only stack where left
  isn't "regress along the chain."

## Item 3 — Stacks live side-by-side on a canvas

**Xian:** These stacks sit **next to each other on a canvas**. Generally
you're focused on one of them, but you should be able to **back up
(zoom out) and swipe over to another one**. And the canvas is where
*multiple top-level decks* live too — the stack for all your health
stuff and the one for all your work stuff might sit on that top-level
canvas.

**What this unifies:** Done/Archive/Trash (Item 2) aren't modal views —
they're neighbors on the same canvas as the Todo deck(s). Navigation is
spatial (zoom out, pan, zoom in) rather than menu-driven. Multiple
decks-by-life-area is the same mechanism, not a separate feature.

**Code-level context:**
- This is the return of the original spatial seed: the 2025 Card Deck
  pivot spec described "spatial thinking" as the foundation, and the
  long-press arc menu was always a stand-in. The audit's gap #6
  ("long-press is the only door to Completed/Settings") is a symptom of
  the views not yet having a *place*.
- Today the app has exactly one deck (one flat task list per storage
  key) and modal views for everything else. Nothing in the TaskStore
  seam assumes one deck, but nothing supports several either.

## Item 4 — Text entry is the wrong tool for this interface (ergonomic concern)

**Xian:** It's going to be very cumbersome to enter text and manipulate
it this way. This might connect to the ideas about populating and
managing the deck automatically (integrations, agents). But — again —
we have to get the basics, the plumbing, and the ergonomics right first
**in the concept model as well** as the code.

**Capture note:** this is registered as a *tension to design around*,
not a feature request: the card interface is optimized for triage
(swipe, flip, focus), and authoring/editing text on it fights the
grain. The auto-population ideas (MCP inbox memo, adapters) are one
possible relief valve but remain parked; the concept model comes first.
