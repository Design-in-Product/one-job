# R1.2 — The Lifecycle Chain: design brief

**Author**: Coral · 2026-07-06 · **Status**: DESIGN for Xian's veto,
build starts next session. Sources: Vision Items 2 & 10 (settled),
Item 15 (block shipped), DOMAIN-MODEL §2b, agreed defaults.

## What we're building

```
                 right →        right →         right → (purge,
 [home deck] ─────────▶ DONE ─────────▶ ARCHIVE ─────────▶ TRASH  confirmed)
             ◀─────────      ◀─────────        ◀─────────
              ← left           ← left            ← left
           (returns HOME,
            top of deck)
```

Right advances a card along its afterlife; left walks it back. Same
two gestures, meaning from location. Nothing is a dead end.

## Settled decisions (from the vision conversation — not reopened)

1. Left from Done = un-done, returns the card **home to the top of its
   deck** (provenance; the existing `uncompleteTask` already does this
   — it becomes the gesture's handler).
2. Right from Done = archive. Archive left = back to Done; Archive
   right = Trash. Trash left = back to Archive.
3. **Trash purges manually only** (the agreed smart default; a timed
   preference can come later). Purge is the ONLY destructive act in
   the app and follows the destructive-action protocol: explicit
   confirm naming the card, no bulk purge in v1.
4. System decks are presented like decks, conceptually distinct but
   not visually exotic (Item 10).

## Design proposals (the parts you can veto)

**A. The Done view becomes a real deck.** The current Completed list is
replaced by the same card experience as everywhere else: top card
face-up, swipe hints reading "← Un-done / Archive →", remaining pile
beneath. Newest-completed on top (recency = relevance in an afterlife).
The "Return to deck" button disappears — the gesture IS the button now.

**B. One view, three rooms.** Long-press → "Completed" becomes
**"Done"**, opening the chain view with a small segmented header:
`Done · Archive · Trash` (counts on each). You swipe cards within a
room; the header moves you between rooms. When the canvas arrives (R2),
these rooms become neighbors in space and the header dissolves — the
chain is built to be *re-homed*, not rebuilt.

**C. Storage stays v2 — no new migration.** Lifecycle state is two new
optional card fields (`archivedAt`, `trashedAt`), fully backward- and
forward-compatible with every existing backup and snapshot. Full
state-is-place document restructuring waits for the canvas era, when
decks become real storage objects. (Gall: the chain must work before
the chain must be beautiful inside.)

**D. Interior cards stay interior.** A sub-card completed inside a
parent stays in the parent's deck (struck-through history, feeding the
Item 15 block); only top-level cards travel the chain. "Any deck feeds
Done" fully arrives with the canvas — noted as a refinement, not
built now.

**E. Undo everywhere.** Every chain gesture gets the same 5-second
undo toast as the main deck. Purge gets confirm-before, not undo-after.

## Build order (test-first throughout)

1. Domain ops: `applyArchive`, `applyUnarchive`, `applyTrash`,
   `applyRestore` (+ purge in the store) with tests.
2. Store methods on the seam (optional, capability-discovered like
   undo/un-complete).
3. Chain view UI (deck-style Done/Archive/Trash with segmented header).
4. e2e smoke: complete → Done deck → archive → trash → walk it all the
   way back home → verify identical card.
5. rc.5 stamp when it feels right on Xian's device.

## Open questions deliberately NOT in this build

Deck-depth material language (Xian exploring) · card aging (Item 22,
same family) · timed purge preference · "any deck feeds Done" ·
lifecycle decks as canvas places (R2).
